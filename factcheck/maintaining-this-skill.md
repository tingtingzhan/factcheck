# Maintaining this skill — correcting errors & editing the guards

Meta-guards that fire at *edit* time, not analysis time.


>[!important] Error-Triggered Skill Repair
>
>When the user's prompt indicates a mistake in a prior response — a correction, "that's wrong", "you misread", a pointed restatement — don't stop at fixing the instance. After correcting against the record, propose an edit to the skill file(s) that would prevent the **class** of error from recurring.
>
>1. **Correct first, against the record.** Restate the right fact with its citation; own the slip plainly, without over-apology.
>2. **Diagnose the class, not the instance.** Name the failure mode (attribute carried across siblings, direction unverified, capacity inferred from non-performance). The instance is one example; the edit must target the class.
>3. **Locate the defect.** Decide whether the record itself was ambiguous/wrong (a _record defect_ — fix the source file) or was correct and misread (a _recall/retrieval slip_ — add a guard or salience aid, not a fact).
>4. **Propose the concrete edit.** Name the file and location; give the exact block to insert or change, in the skill's callout idiom. Prefer a **general guard** when the error is a reasoning class likely to recur; add a **local contrast/caution** only to make one specific asymmetry glanceable.
>5. **State the limit honestly.** A salience aid helps only if actually re-read at analysis time; the load-bearing fix is usually a guard phrased as a _step to run_, not a note that hopes to be noticed.
>
>**Calibrate — don't manufacture a defect.** If the source was already unambiguous and correct, say so: the fix is "read the item at hand; don't pattern-complete," not a new rule. Not every one-off correction earns a permanent guard — reserve one for a **class** likely to recur, since low-value guards dilute the skill.
>
>Do **not** edit an `.md` skill file in place. Deliver the patched file — or the exact diff — for the user to apply.


>[!important] Keep Conversational Additions Skill-Agnostic
>
>When a guard is added or edited through conversation, verify the wording carries no context from any other skill co-invoked in that conversation. factcheck guards are scenario-agnostic and portable; a live discussion is anchored in some record, and its specifics leak easily into the phrasing.
>
>Before committing an addition, strip every non-general element:
>- named parties, roles, or characters from the worked example;
>- domain-, artifact-, or fact-specific nouns pulled from the record at hand;
>- any premise true only of the co-invoked skill's setup, not of records in general.
>
>Keep the worked example in the conversation; let only the abstracted reasoning class enter the skill file. Test each clause: would it read correctly against an unrelated record? If it needs the originating scenario to make sense, it is not yet a guard — re-abstract it.
